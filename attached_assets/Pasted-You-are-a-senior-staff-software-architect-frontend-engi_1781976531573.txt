BACKEND RESPONSIBILITIES

1) Health and system status
- Create a health endpoint.
- Create a model status endpoint.
- Detect whether the ML weights are present.
- Detect whether the backend is running in fallback mode.
- Return backend readiness and device info.

2) NetCDF dataset inspection
- Accept a .nc file.
- Use xarray to inspect the file.
- Detect variables, dimensions, coordinates, attributes, time steps, dtype, min/max, and candidate display channels.
- Do not assume any one variable name.
- Rank likely visual variables intelligently.
- Return structured metadata to the frontend.
- If multiple candidates exist, allow the frontend to choose one.
- If the file has unusable variables, explain why.

3) Scientific preprocessing
- Convert selected NetCDF scientific data into image-ready frames.
- Normalize values into 8-bit image space carefully.
- Preserve contrast and structure.
- Support grayscale preview.
- Optionally support false-color preview if helpful.
- Save preview images in a dedicated folder.
- Log preprocessing steps clearly.
- Handle corrupted files, missing dimensions, NaN/inf values, and invalid arrays gracefully.

4) Frame extraction logic
- If the dataset contains a time axis, support selecting two time slices or a time window.
- If only one file is available, support previewing the dataset and selecting two interpretable frames or slices.
- If the user uploads two preprocessed frames directly, support that too.
- Keep the interface flexible and robust.

5) ML model adapter
- Create a model adapter layer separate from routes.
- The adapter must support a RIFE-family interpolation checkpoint as the default concept.
- Structure the adapter so that another interpolation model could be swapped in later without changing the frontend or API.
- Detect CPU/GPU automatically.
- Load weights from backend/models.
- If the checkpoint is missing, switch to a safe fallback demo mode that produces a simple blended interpolation so the app still works.
- Clearly mark fallback mode in the API response and frontend UI.
- Put explicit comments where the real model weights should be placed.
- Keep model loading isolated, clean, and replaceable.

6) Async job flow
- Do not make the app feel frozen during long processing.
- Use a job-based flow for interpolation.
- Create a job when processing starts.
- Return a job ID.
- Allow polling or live updates for status.
- Support progress updates such as:
  - uploading
  - inspecting
  - preprocessing
  - loading model
  - running inference
  - generating validation
  - completed
- The job system can be lightweight, but it must exist.

7) Progress updates
- Add WebSocket or Server-Sent Events support if feasible.
- If real-time streaming is too heavy, provide a clean polling endpoint that updates progress.
- The UI should show live processing stages, not just a spinner.

8) Validation and scientific trust
- If a ground-truth midpoint frame is available, compute useful metrics such as:
  - SSIM
  - PSNR
  - MAE
  - RMSE
- If ground truth is unavailable, do not fake metrics.
- Instead, show preprocessing stats, inference mode, and a technical explanation.
- Add a scientific explainability object that describes what the system did.

9) Explainability payload
Return a structured explanation such as:
- selected variable
- detected dimensions
- normalization method
- interpolation mode
- fallback vs real model
- available confidence indicators
- notes about dataset limitations

API CONTRACT
Generate these endpoints:

GET /health
- status
- version
- timestamp
- model availability
- fallback mode
- backend readiness

GET /api/models/status
- model file exists or not
- model path
- inference readiness
- fallback mode
- device info

POST /api/inspect
- accepts one .nc file
- returns dataset metadata, candidate variables, dims, coords, and inspection explanation

POST /api/jobs/create
- creates a processing job
- accepts upload info and selection info
- returns job ID and initial status

GET /api/jobs/{job_id}/status
- returns progress, stage, messages, and completion state

GET /api/jobs/{job_id}/result
- returns original frame paths, interpolated frame path, metadata, metrics, explanation, and output links

POST /api/interpolate
- accepted as the main orchestration endpoint if you want a simpler direct path
- it may internally create and run a job
- should still return structured output
- do not make the frontend depend on a fake synchronous completion only

If you implement both job-based and direct orchestration endpoints, keep them consistent.
